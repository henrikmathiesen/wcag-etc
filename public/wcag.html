Accessibility concerns #

It is estimated that 1/5 of the population has some sort of disibility; vision impairment, body motor impairment or cognitive issues. There are also "situational disabilities" like weak internet connection, a small viewing device, an old device/browser, bright light or a dark envirionment, and also being in a stressfull situation.

Accessibility is the practice of making a site or an app usable for the widest range of people. We can not assume that everybody uses an interface the way we do. Some people are blind or partially blind and use screen readers, some people are color blind, some people use only the keyboard for navigation because using a mouse is difficult for them.

Making an interface accesable for everyone is "the right thing to do", it also means that more people can use your site which could mean more profit, it also helps with search enginge optimazation. And in some countries it is actually the law.

Planning for accessibility #

  • Have it in mind from the beginning
  • Set up goals, get everyone on the same page
  • Put someone in charge of accessibility
  • Evaluate tools and frameworks
  • Do research on best practices
  • Create different user personas

Designing for Accessibility #

Responsive design

Making apps and sites respond to different viewports is a part of accessibility. There are also other things to consider, like color contrast helping users in a bright envirionment and simplistic interfaces helping people in a busy envirionment.

Color

1 in 12 men and 1 in 200 woman are color blind, that means roughly 5% of the population. Color blindness is a complex matter, but it breaks down to people having difficulties distinguish between colors. The most common is red/green deficiencies.

The take away is, dont rely soley on color to convey messages and design hierchy. Complement it with text or symbols.

Also take into considoration that different display displays colors differently, for example an high quality monitor might distinguish our colors fine but the colors looks almost the same on a low end monitor. Also the lighting in the environment impacts colors. Contrast helps!

Typography

It is very important. The characters should be easely distinguishable but not too large. Also think about color contrast and contrast between bold and normal.

A good body copy is 16-20px, but better to convert them to REM with SASS, Zurb Foundation has such a built in mixin, rem-calc();

Line length should be 45-75 characters. Line height should be about 125 procent of text size. Justified text should be avoided. Avoid ALL CAPS.

Form Design

Clearly communicate required fields and formats rules (do not soley rely on color). Every form field should have an assosiated label (label for input id), do not use placeholder as label (since its lost on focus) but placeholder can be used to examplifie format. Communicate errors, and again, do not soley rely on color.

Touch Targets

Design the touch target size to minimum about 45 x 45 px, so it is easy to hit with a finger tip. There should also be enough space between touch targets.

Focus and Active Indicators

Some users use only the keyboard for navigation, they need a clear indikator of where focus is on the page. Browsers implements focus indikation behavior so do not override that and do not hide the indikators with css.

Motion

Some people experience motion sickness and similar because of animations. Avoid flashy design. Could also implement a feature where users can turn off animations.

Designing for Crisis

Apps and sites are often designed to fit the ideal user, a happy and calm person with all the time in the world. That not always the case. We should design with crisis driven personas in mind. The argument is that if a person in a crisis sitation can navigate our site then everyone can as well.

Coding for Accessibility #

Proper code is at the heart of our accessibility efforts. Key points here are: proper semantics, dont use text as images, add alt attributes for images, accessible from a keyboard, performance.

Proper document structure

Use the right tag for the job. If we remove the CSS, the page should still render in an logical order and with a clear hierchy. h1-h6 are critical for hierchy, they show how sections relate to one another and also provide a target that screen readers can use to jump from section to section. h1-h6 should be used in the proper order, h3 for example should have a h2 parent. Unordered lists, ordered lists and description lists are also critical to the document hierchy, use description lists when structure for description are needed (see list examples).

Use semantic that makes navigating the interface with a keyboard easier. For example if the interface is having a main navigation at the top, a side navigation at the left and main content in the middle, the tab order should be first the main navigation then the main content and the side navigation last.

Landmark elements

Use these so users can jump between sections.

  • header
  • nav
  • main (can only be one)
  • footer
  • aside

<nav class="site-navigation" aria-label="site"/>
<nav class="page-navigation" aria-label="page"/>

Images

Images that are used purely for design and do not communicate content to the user can be used as CSS backgrounds or give it, and empty alt attribute. However images in content that does communicate something to the user must be handled with accessibility in mind, they need to have an alt attribut that provides a description, keep the description short and to the point, it should communicate what the image is trying to say.

For more complex images, like a diagram, we can provide a long description (longdesc="graph.txt"), the value of the attribute can be a text file, a link to another page or a link to a section on the same page (it is ok to hide it with display:none). However, the browser support longdesc is not good at this point (sep, 2015), therefor it is best to provide a plain old link to the long description. To do this right we need an aria attribute.

aria-describedby

<img src="graph.jpg" aria-describedby="graph-desc" alt="Graph" />
<a href="graph-desc.html" id="graph-desc">Get Graph Details</a>

Content

All your users, no mather their profile, came for your content and they should be able to get it. Place a Skip to main content link at the top of the page, when using audio, provide a transcript for people that have hearing issues, when using videos provide caption. Also dont use links out of context, dont write "Read more", instead write "Read more about our products", if it is crucial to the design that links say "Read more" we can insert it with a:after {content: "Read More"} in css and hide the "Read more about our products" (dont hide it with display none, see below).

HTML Attributes

To make our code accessible, there are some HTML attributes that we need to be aware of.

  • Provide the proper language attributes for the html tag, html lang="en"
  • If text content in another section of the page have a different language, provide lang attribute for that tag also
  • Provide alternative text for image, as described above
  • When using abbreviations, provide a title attribute showing the full word
  • When using forms, use label for with a matching id
  • When using forms, use the required attribute for required fields
  • Use tabindex="0" if using non tabable controls like a span, 0 means tab order is in normal document flow
<html lang="en">...</html>
<blockquote lang="fr">...</blockquote>
<abbr title="Hyper Text Markup Language">HTML</abbr>
<form>
<label for="myform-firstname">First Name</label>
<input type="text" id="myform-firstname" />
</form>

Forms

Forms can be a little complex, but we need to make them simple and structure them in a logical order. We accomplish that by using fieldset and legends, labels with for attribute and inputs with id matching the for attribute.

Use the proper semantic for the type of input, for example input type="email".

Use the required attribute when an input is required.

Dont use placeholder as a replacemnt for a label. Placeholder can however be used to help the user with formatting examples.

Add helper text where needed, dont rely on the placeholder.

Make sure that the taborder works (it should work if the form is structured in a logical way).

When a label is not needed, visually, because of the context, lets say for a search bar at the top of the page, we still need to provide a label so users with screen readers can understand the context. We can hide the label with CSS. Use the proper CSS to hide it, do not use display:none since it will hide it from screen readers as well. Instead use .hidden { position:absolute;left-99999px; }

See example of form.

Keyboard Control

This area of accessibility is easely testable, simply stop using the mouse and see if we can use the interface.

Examples of users would benefit from keyboard controls: blind people, people with fine motor control issues, people with a broken hand.

Make sure to not remove the focus indicator (input:focus{}). Also keep in mind that display:none will completly hide the element, both visually and also from assistive technoligy.

Tab order is another area of concern. The flow of keyboard navigation should be logical, left to right and top to bottom. However there are exceptions for layout.

Dont use div or span for clickable items (they will not have a tabindex), instead use a and button. However div or span can be given a tabindex by adding tabindex="0" (where 0 means the natural document flow).

Dont play around with custom tabindex (tabindex=2,3 etc), it can become a maintenance nightmare.

CSS and Accessibility

User styling will override our styles, users can for example change font, font size and contrast. It is important that the content looks good under these circumstances, for example a button expanding to fit the bigger font size. Dont use pixel values, use relative units instead like REM or percent.

Dont rely on CSS alone to present meaning, use the proper semantic (like h1-h6 for headers).

Sometime we want to hide content visually but make it accesable to screen readers, for example a header saying "Main Navigation". We can hide this content with (clip property has good browser support, but is now deprecated): 

          
            position: absolute !important;
            width: 1px;
            height: 1px;
            overflow: hidden;
            clip: rect(0, 0, 0, 0);

This is another way to do it: 

        
          position: absolute !important;
          top: auto !important;
          left: -9999px !important;
          width: 1px;
          height: 1px;
          overflow: hidden;

Twitter Bootstrap has a CSS class named 'sr-only' and Foundation has a 'show-for-sr', for this purpose.

Note that display:none or visibility:hidden or width:0;height:0; will hide the element from screen readers as well.

Performance

A slow internet connection can cripple the user experience if the app has many http requests and big files to download. Try to make fewer http request by bundle javascript and css files. Also use compression on images, see tools.

Usage of image sprites is good for reducing the number of http requests and caching (every image is cached at once). There are Gulp plugins for this, just google it.

SVG is a good option for its vector properties (scalable).

Critical CSS is a concept where we place CSS in header to prevent http request for external CSS, this "critical CSS" is used for styling the "above the fold" content, so users get it as quickly as possible. Can automate it.

Progressive Enhancement

Instead of designen and coding for users with modern desktop machines with the latest browsers, instead target the low end users. Write well structured HTML, add a layer of CSS, and a layer of JavaScript. The app should work with just the HTML, and with HTML + CSS.

ARIA Roles and Attributes #

This is new stuff for me, so it gets its own section.

What is it?

  • WAI-ARIA = Web Accessibility Initiative - Accessible Rich Internet Applications
  • A specification developed by the W3C
  • Helps bring accessibility to modern sites and apps using JavaScript and Ajax
  • With ARIA, custom controls can be marked as widgets of different types

There are 2 ARIA categories: Roles and States and properties.

Roles

Roles define what a widget is or what it does. Example of roles are: menus, sliders, progress meters, modals. Roles also describe structure like: headings, regions, tables.

States and properties

State describes the current interaction state of the widget, examples: checked, disabled, selected, hidden.

Properties describe the charactaristic of the widget, like: drag n drop, live regions (regions that will be updated), modals (that will open)

Prefer the correct semantic over ARIA

If there is a semantic correct element for the job then use it, and it is often uneccessary to use ARIA in that case. But if we must use sliders whith only divs and spans we can use ARIA semantic to better help assistive technoligy undertand what it is. Here is some good info to read.

Testing, Resources and Tools #

Accessibility is about the human experience. It is ideal to test with many users that matches the personas we set up in the planning phase. An easy way to test for accessibility is to simply stop using a mouse. There are a number of resources and tools to help us make our sites and apps more accessible.

WCAG 2.0 from W3C

Intended for Web developers, Web designers, Web Editors, Content Management System developers, Web accessibility tool developers.

There are 12 guidelines devided into 4 areas (POUR): Perceivable, Operable, Understandable, Robust. It has has 3 levels of compliance: A (minimum), AA (Mid-Level), AAA (Highest Level).

According to WCAG 2.0 it is allowed to use JavaScript as long as the success critera are met.

Perceivable

Users must be able to perceive the information. We need to provide text alternatives for any non text content. We also need to make the content adaptable so it can be presented in different ways, for example larger text or read by a screen reader. We need to think about color contrast and typography, use more than color to convey meaning, make it easy to see and hear content and provide audio controls for audio longer than 3 seconds.

Operable

Users must be able to operate all user interface controls. Make the interface keyboard accessible, use links and buttons properly, wire up events with JavaScript and add ARIA roles and attributes, use logical form structure. Provide the user enough time to understand and use the interface. Dont cause seizures, avoid anything that flashes more than 3 times a second. User proper semantic for heading. Add "Skip to Main Content" link. Tabable controls (links, buttons, form controls) should have the proper focus and active state.

Understandable

Users must be able to understand the information and user interface. Add the proper lang attribute to the HTML tag and to any sub section that contains a different language. Add abbreviations with title attributes so users can get the full meaning of the abbreviation. Make the interface appear and operate in predictible ways. Make forms easy to fill out and help the user with validation.

Robust

The site should work in a wide variety of browsers and assistive technoligy. Add ARIA when needed.
